# Technical Documentation T3S3 Ranging Mesh System

## 1. System Overview

This system is designed for local robot positioning ( coordinates) using LILYGO T3S3 (SX1280) boards. It utilizes a Time-of-Flight (ToF) approach to determine distances between nodes.

 Technology LoRa 2.4 GHz (Semtech SX1280).
 Method Hardware-accelerated Ranging.
 Nodes 3 Stationary Anchors, 1 Mobile Tag (Robot).
 Precision 0.5 – 1.5 meters.

## 1. EXECUTIVE SUMMARY

The **Turbo Interceptor** is a high-performance quadrupedal robot platform designed for research in inverse kinematics and real-time locomotion. Unlike traditional Arduino-loop based robots, this system utilizes a **Real-Time Operating System (FreeRTOS)** to decouple physics calculations from network latency. The system features a custom "Creep Gait" algorithm ensuring 3-point stability at all times and a bi-directional telemetry system that visualizes the robot's physical state in a 3D web interface.

---

## 2. HARDWARE ARCHITECTURE

### 2.1 Component Specification

* **MCU:** Espressif **ESP32-C6** (RISC-V Single Core, 160MHz). Selected for WiFi 6 capabilities and high-efficiency processing.
* **Actuators:** 12x **MG996R** High-Torque Servo Motors (Running at 50Hz PWM).
* **Servo Controller:** **PCA9685** 16-Channel 12-bit PWM Driver (I2C Interface).
* **IMU (Optional):** MPU6050 (Currently bypassed in v50.5 for stability testing).
* **Power System:** Dual-rail architecture (Separate logic and servo power) is recommended to prevent Brown-out Resets (BOR).

### 2.2 Wiring Diagram (Pinout)

The system utilizes the I2C bus for peripheral communication.

| Component | ESP32-C6 Pin | Protocol | Notes |
| --- | --- | --- | --- |
| **PCA9685 SDA** | GPIO 6 | I2C | 400kHz Fast Mode |
| **PCA9685 SCL** | GPIO 7 | I2C | 1000ms Timeout Enabled |
| **Logic Power** | 3.3V / GND | DC | Stabilized source |
| **Servo Power** | External | DC | 6.0V - 7.2V High Current |

---

## 3. SOFTWARE ARCHITECTURE (RTOS)

The firmware moves away from the "Super-Loop" paradigm to a thread-safe, multi-tasking architecture.

### 3.1 Task Scheduling

The CPU time is sliced between two primary tasks with strictly defined priorities:

1. **`Task_Physics` (Priority 5 - HIGH/REAL-TIME)**
* **Frequency:** 50Hz (Deterministic).
* **Responsibility:** Calculates Inverse Kinematics (IK), generates gait trajectories, interpolates movement, and writes to PCA9685.
* **Constraint:** Must never be blocked by network operations.


2. **`Task_Network` (Priority 1 - LOW/BACKGROUND)**
* **Frequency:** Asynchronous.
* **Responsibility:** Handles WiFi connection, WebSocket packets, JSON parsing, and serves the HTML/JS Interface.
* **Constraint:** Yields CPU to Physics task immediately upon interrupt.



### 3.2 Thread Safety (Mutex)

To prevent "Race Conditions" (where the network writes target coordinates while physics is reading them), a **Binary Semaphore (Mutex)** is implemented.

* *Protocol:* Any task accessing `RobotState` must acquire the `dataMutex`.
* *Timeout:* 5-10ms max wait time to prevent deadlocks.

---

## 4. KINEMATICS & LOCOMOTION

### 4.1 Mechanical Model (Leg)

Each leg consists of 3 links (Degrees of Freedom):

1. **Coxa (Hip):** Horizontal rotation.
2. **Femur (Thigh):** Vertical lift.
3. **Tibia (Shin):** Vertical extension.

The math engine calculates the required servo angles using trigonometry (Law of Cosines) based on a target coordinate  relative to the leg mount point.

### 4.2 Gait Engine: "Stable Creep"

Version v50.x implements a Wave Gait sequencer:

* **Stance Phase:** 3 legs are on the ground, moving backward relative to the body to propel the robot forward.
* **Swing Phase:** 1 leg lifts (Sine wave trajectory), moves forward, and places down.
* **Sequence:** `FL`  `BR`  `FR`  `BL`.

---

## 5. COMMUNICATION PROTOCOL (API)

Communication is handled via WebSockets (Port 81) using JSON payloads.

### 5.1 Inbound Commands (Client  Robot)

| Command | JSON Structure | Description |
| --- | --- | --- |
| **Move** | `{"cmd":"pad", "lx":0.5, "ly":1.0...}` | Joystick input (normalized -1.0 to 1.0). |
| **Mode** | `{"cmd":"mode", "val":"walk"}` | Switch state machine (Stand / Walk / Manual). |
| **Reset** | `{"cmd":"reset"}` | Force coordinates to (0,0,0) and Mode to Stand. |
| **Height** | `{"cmd":"h", "val":-60}` | Adjust body height offset (). |

### 5.2 Outbound Telemetry (Robot  Client)

The robot broadcasts its state for the 3D visualization:

```json
{
  "p": 0.0,  // Pitch (from IMU/Sim)
  "r": 0.0,  // Roll (from IMU/Sim)
  "l": [     // Leg Arrays [Coxa, Femur, Tibia, Twist]
    [0, 45, -90, 0], // FL
    [0, 45, -90, 0], // FR
    ...
  ]
}

```

## 6. DEPLOYMENT & SETUP

### 6.1 Prerequisites

1. **IDE:** VS Code (PlatformIO) or Arduino IDE 2.x.
2. **Libraries:**
* `Adafruit_PWMServoDriver`
* `ArduinoJson` (v7)
* `WebSockets` (Markus Sattler)


3. **Board Definition:** ESP32-C6 Dev Module.

### 6.2 Flashing Procedure

1. Set `WIFI_SSID` and `WIFI_PASS` in `config.h`.
2. Ensure robot is powered by **USB only** (High voltage disconnected) during flash.
3. Upload firmware.
4. Open Serial Monitor (115200 baud) to verify `I2C SCAN COMPLETE`.

---

## 7. TROUBLESHOOTING

| Symptom | Probable Cause | Solution |
| --- | --- | --- |
| **"NO I2C DEVICES FOUND"** | Wiring / Power | Check SDA(6)/SCL(7) connection. Ensure PCA9685 has 3.3V logic power. |
| **Robot freezes when walking** | Noise / Voltage Sag | Add capacitors (1000uF) to servo power rail. Twist servo cables. |
| **3D Interface empty** | WebGL Error | Ensure browser supports hardware acceleration. Check telemetry JSON in console. |
| **Legs move randomly** | Calibration | Check `CFG` array signs and offsets in `setup()`. |
